# 이메일 수집 프로그램 리서치 노트

## 1) 프로젝트 목적과 현재 상태

이 프로젝트는 `customtkinter` 기반 데스크톱 앱으로, 로그인/회원가입(구글 시트 기반 멤버십) 이후 두 개의 수집 소스(`Daum`, `잡코리아`)에서 업체 연락처 데이터를 수집해 엑셀로 내보내는 구조다.

핵심 특징:
- 플랜 기반 기능 제한(무료/기간제/영구)
- 탭 기반 병렬 작업 UX (Daum 내부 다중 탭)
- Selenium 기반 크롤링 + 간단한 휴리스틱 필터링
- 최근 추가된 공용 금지목록(BlockList)으로 전체 탭 차단 정책 공유


## 2) 파일별 책임(Responsibility Map)

### `main.py`
- 앱 엔트리포인트
- 로그인/회원가입 UI + 인증 흐름
- 버전 체크(`LatestVersion`, `UpdateLink`)
- 플랜/만료 상태 반영
- `IntegratedExtractorApp`에서 공용 `BlockList()` 1회 생성 후 탭에 주입

### `database.py`
- Google Sheet 연결 단일 책임
- 전역 `sheet` 객체 초기화
- 연결 실패 시 `None` 반환

### `ui_daum.py`
- Daum 검색 결과에서 후보 사이트 수집
- 각 사이트 본문/하위 링크에서 이메일+전화번호 추출
- 테이블 렌더, 실행/중지/초기화/엑셀 저장
- 금지목록 입력/엑셀 불러오기/초기화 UI 제공
- 수집 직전 `blocklist.should_block(...)`로 차단

### `ui_jobkorea.py`
- 잡코리아 리뷰 리스트에서 기업 후보 수집
- 기업정보/홈페이지 이동 후 이메일 추출
- 테이블 렌더, 실행/중지/초기화/엑셀 저장
- 금지목록 UI(입력/불러오기/초기화) 제공
- 수집 직전 `blocklist.should_block(...)`로 차단

### `blocklist.py`
- 공용 차단정책 모델
- 업체명/도메인/이메일 세트 관리
- 스레드락 + 변경 구독자 알림
- 엑셀 로더 + 정규화/매칭 로직 캡슐화


## 3) 런타임 데이터 흐름

1. 앱 시작 시 `database.sheet` 초기화 시도  
2. `LoginWindow.login()`에서 시트 사용자 목록 조회  
3. 인증 성공 시 `IntegratedExtractorApp` 실행  
4. `IntegratedExtractorApp`가 `BlockList` 1개 생성 후 Daum/잡코리아 탭에 주입  
5. 각 탭에서 수집 시작:
   - 백그라운드 스레드에서 Selenium 크롤러 실행
   - UI 갱신은 `self.after(...)`로 메인 스레드에 전달
6. row 생성 직전에 `blocklist.should_block(company, url, email_str)` 실행  
7. 차단 불일치 데이터만 `data_list`와 `Treeview`에 반영  
8. 저장 시 `pandas.DataFrame(...).to_excel(...)` 실행


## 4) 공용 금지목록 설계 이해

공용화 방식:
- `main.py`에서 `self.blocklist = BlockList()` 생성
- `DaumTabUI(..., blocklist=self.blocklist)`, `JobKoreaTabUI(..., blocklist=self.blocklist)` 주입
- Daum 내부 `추출 1~5` 인스턴스도 동일 객체 참조

매칭 규칙:
- 업체명: 공백 제거 + 소문자 후 부분일치
- 도메인: `www.` 제거한 exact match
- 이메일: exact match
- 복수 이메일 문자열(`a@x.com, b@y.com`)은 분리 후 개별 검사
- 이메일 도메인도 금지도메인과 교차 검사

UI 동기화:
- 각 탭이 `blocklist.subscribe(...)`로 변경 알림을 구독
- 한 탭에서 추가/초기화/엑셀 로딩 시 다른 탭 카운트도 자동 갱신


## 5) 코드 품질 관찰 (강점)

- 책임 분리가 비교적 명확함(인증/DB/소스별 크롤러/차단정책)
- UI 스레드 안정성을 위해 `after()` 사용
- 무료 플랜 제한 로직이 사용자 흐름에 잘 녹아있음
- 공용 금지목록이 별도 모듈로 추출되어 중복 로직이 줄어듦


## 6) 리스크/취약점 (실무 관점)

### A. 광범위한 `except: pass`
- 크롤링 안정성은 높이지만, 원인 추적이 거의 불가능
- 실제 운영에서 사이트 DOM 변경 발생 시 “조용한 실패”로 끝날 가능성 큼

### B. 크롤링 선택자/텍스트 의존도
- Daum/잡코리아 DOM과 문구에 강하게 의존
- 작은 레이아웃 변경에도 수집량 급감 가능

### C. 성능/반응성
- `time.sleep(...)` 기반 대기 사용
- 느린 사이트에서 소요 시간이 길어지고 체감 품질 하락 가능

### D. 리소스/수명주기
- UI 파괴(destroy) 이후 예약된 `after` 콜백이 남는 경우 Tcl 경고가 발생할 수 있음
- 종료 시점의 콜백 정리 전략이 현재는 약함

### E. 보안/운영
- 인증정보(`credentials.json`) 파일 의존
- 실행 위치/배포 시 경로 관리가 중요
- 사용자 비밀번호가 시트에 평문 구조로 보관될 여지가 있음(운영 정책 검토 필요)


## 7) 개선 우선순위 제안

1. **로깅 표준화**  
   `except: pass` 일부를 `log_cb("...")` 형태로 교체해 장애 관측성 확보

2. **선택자 내구성 강화**  
   우선순위 선택자 체인 + fallback 체계화

3. **UI 종료 안정화**  
   destroy 시 `after_cancel` 가능한 핸들 관리 검토

4. **금지목록 영속화**  
   앱 재시작 후에도 유지되도록 JSON/Excel 자동 로드 옵션 추가

5. **인증 데이터 보안성 재검토**  
   최소 해시 저장/관리자 승인 파이프라인 분리 검토


## 8) 실제로 이해한 핵심 포인트 요약

- 이 앱의 본질은 “회원제 B2B 연락처 수집 툴”이며, UI는 단순한 입력 폼이 아니라 플랜 정책을 강하게 반영하는 운영 도구다.
- Daum은 “사이트 후보 -> 본문/하위링크 탐색 -> 이메일/전화 추출”의 2단 수집, 잡코리아는 “리뷰 리스트 -> 기업정보 -> 홈페이지 -> 이메일 추출”의 경로 기반 수집이다.
- 최근 기능의 본질은 “탭별 개별 차단”이 아니라 “프로세스 전역 차단 정책”으로의 승격이며, `BlockList` 주입 구조로 이를 달성했다.
- 현재 품질에서 가장 중요한 다음 단계는 기능 추가보다 “관측성(로그) + 내구성(선택자/예외)” 강화다.
